# CLAUDE.md — Healthcare RCM Intelligence Platform

Read this file completely before doing any work. Every agent operating in this
repository is bound by the rules below. When rules here conflict with a task
prompt, ask the human before proceeding.

## 1. What this project is

An end-to-end healthcare revenue-cycle analytics platform built as a portfolio
project. It ingests **official CMS synthetic Medicare claims** into a PostgreSQL
warehouse, adds a **clearly labeled simulated adjudication layer** (denials,
appeals, workflow events, payments timing), computes RCM KPIs, trains
explainable ML models (pre-submission denial risk, appeal success, expected net
recovery prioritization), and serves results through a FastAPI scoring API and
a Streamlit analyst dashboard.

**The single most important property of this project is honest data
provenance.** No simulated value may ever be presented as real. This is what
makes the project credible in interviews.

## 3. Data provenance rules (NON-NEGOTIABLE)

1. Every field in every curated table carries a classification:
   `SOURCE` (from CMS files unmodified), `DERIVED` (computed from SOURCE),
   `REFERENCE` (from official code/facility reference files), or
   `SIMULATED` (generated by our simulation layer).
2. Every simulated table and column name is prefixed `sim_`.
3. The provenance register (`docs/provenance_register.md`) and the data
   dictionary (`docs/data_dictionary.md`) must be updated in the same PR that
   adds or changes any table or column.
4. **Simulated linkage rule:** synthetic claims contain synthetic provider and
   facility identifiers. They do NOT join to real NPPES NPIs or real hospital
   CCNs. Enrichment happens through a reproducible, seeded random assignment
   of claims to real facilities/providers (stratified by state and type). The
   crosswalk table is itself classified SIMULATED.
5. **Payer rule:** Medicare FFS has one payer. Our multi-payer dimension is
   100% simulated. Every payer-level analysis, view, and dashboard page must
   carry the simulated-data banner.
6. Raw CMS files are never modified and never committed to git. Only download
   scripts, manifests, and checksums are committed.
7. No CPT descriptions in the repo (AMA-licensed). HCPCS Level II public
   descriptions are permitted. Denial reason taxonomy uses CARC codes as
   category labels only.

## 4. ML leakage rules (NON-NEGOTIABLE)

Model A (pre-submission denial risk) predicts at the moment BEFORE claim
submission. The forbidden-column blacklist lives in `config/model.yaml` under
`forbidden_features`. It includes, at minimum:

- `sim_denial_flag`, `sim_denial_reason`, `sim_denial_category`
- `sim_allowed_amount`, `sim_paid_amount`, any payment or adjustment field
- adjudication dates, appeal fields, recovered amounts
- all `sim_workflow_events` occurring after submission timestamp
- the latent generator probability (`sim_latent_p`) — validation only, never a feature

Rules:
1. `tests/leakage/` contains an automated test that fails the build if any
   forbidden column (or a column derived from one) enters a training matrix.
2. All historical-rate features (provider clean-claim rate, payer denial
   propensity) must be computed with prior-period or out-of-fold logic only.
3. Temporal splits, never random splits, whenever time-dependent features exist.
4. Preprocessing is fit on training folds only.
5. The ml-engineer agent must NOT read `src/simulation/` internals. It consumes
   the feature store as if the data were real. (This separation is deliberate
   and is part of the interview story.)

## 6. Workflow conventions

- Work happens on feature branches; `main` must always pass `make test`.
- Every PR updates `tasks.md` (move item, note blockers).
- Commit messages: `[agent-name] short description`.
- Every new module gets tests in the matching `tests/` subfolder in the same PR.
- Secrets only in `.env` (gitignored). `.env.example` documents required vars.
- SQL style: lowercase keywords ok, snake_case identifiers, every view starts
  with a comment block stating grain, sources, and provenance classification.
- Python: ruff for lint/format, pytest for tests, type hints on public APIs.
- Dashboard: every page renders the synthetic-data banner component. No page
  ships without it.

## 7. Definition of done (per phase)

- **Data**: checksums + manifest recorded; schema tests pass; row counts
  reconcile to source within documented tolerance.
- **Warehouse**: FK integrity passes; views reconcile to control queries.
- **Simulation**: same seed ⇒ identical output; directional validity checks
  pass (missing auth raises denial probability, etc.); assumptions documented
  in `docs/assumptions.md` with citations to published industry benchmarks.
- **ML**: baseline vs advanced comparison reported; calibration plot produced;
  leakage tests pass; slice metrics by payer/facility/service line reported.
- **App**: `docker compose up` works from clean clone; API schema validated;
  dashboard totals reconcile to SQL control queries.
- **Docs**: README, data dictionary, model card, assumptions current.
